Eighth beta release of Loom — an evidence operating system for civil-rights legal teams that turns mixed field evidence (video, photos, audio, FOIA returns) into a defensible event timeline where every claim traces back to source material. Civil-liberties tooling: no face recognition, no suspicion scoring, no automated identity resolution.

> **Beta scope:** prepared for first use in Illinois (state and federal § 1983 actions). Defaults are documented in [`docs/requirements.md`](docs/requirements.md) and are jurisdiction-tunable.
>
> **Not legal advice.** Statutes and rules of evidence are referenced for design context only. Admissibility, consent, retention, and disclosure decisions for any specific matter belong with licensed counsel.

## Release focus

v0.1.9 fixes a desktop bug that blocked first-run onboarding outright: a fresh Loom Desktop Lite install could never create its admin account. The app would prompt for a data directory, accept it, then reject account creation with "this install is already set up" and bounce back to the data-directory screen — an inescapable loop with no way into the app. The fix is a one-expression correction in the backend first-run handler; the rest of the release is the weekly Dependabot wave landed alongside it.

## Highlights — desktop first-run loop

- **First-run account creation rejected on every install (`backend/src/loom/api/v1/first_run.py`).** `POST /first-run/complete` creates the bootstrap admin with a single atomic `INSERT … SELECT … WHERE NOT EXISTS`, so two racing requests can't both create an admin. The guard double-wrapped its existence check: `user_exists` was already an `EXISTS` construct (`select(…).exists()`), and it was wrapped in `exists()` again before negation, compiling to `WHERE NOT (EXISTS (SELECT EXISTS (SELECT 1 FROM users)))`. The inner scalar subquery always returns exactly one row, so the outer `EXISTS` was always true and the guard always false — the insert wrote **zero rows even on a fresh, empty database**, and the endpoint returned `409 Conflict` unconditionally. The frontend derives its `first-run ↔ login` redirect from `/first-run/status` (which correctly reported "no users yet"), so the contradictory 409 produced the redirect loop. Corrected to negate the `EXISTS` directly (`.where(~user_exists)`). No data migration or factory reset is needed: because the insert always failed, no admin user was ever created, so onboarding completes the first time after the upgrade.
- **Regression coverage closed.** The first-run unit tests exercised a mock session that hard-coded the insert's `rowcount`, so they never compiled the real statement and could not catch this. A new integration test (`backend/tests/integration/test_first_run_complete_sql.py`) drives `/first-run/complete` against a real on-disk SQLite database bootstrapped exactly the way the sidecar boots it — asserting empty-db → `201`, second call → `409`, and that `/status` flips to `first_run_required: false` after success.

## Also addressed — security advisories

- **js-yaml quadratic-complexity DoS (merge-key handling).** Pinned forward to `js-yaml >= 4.2.0` via a `pnpm` override; js-yaml is a transitive dev-tooling dependency.
- **esbuild Deno-installer RCE (GHSA, `NPM_CONFIG_REGISTRY`) — assessed not applicable.** The advisory's vector is esbuild's Deno installation module, which downloads its binary without integrity verification. Loom consumes esbuild only as a build-time-only `devDependency` via `pnpm`, with lockfile-pinned, integrity-checked per-platform binaries — the Deno install path is never invoked, and esbuild is never shipped to users. The code-level remediation (esbuild ≥ 0.28.1) is incompatible with the current Vite 6 toolchain (Vite 6 pins `esbuild ^0.25`; forcing 0.28.1 breaks the production build) and is tracked with the Vite 8 upgrade. The alert is dismissed with this rationale.

## Migrations

- No new Alembic revisions in v0.1.9. No schema, data, or configuration changes. Existing v0.1.8 desktop installs upgrade in place; the only runtime-affecting change is the first-run handler fix (which only takes effect on a not-yet-onboarded install) plus the upgraded library versions, all validated by the existing test, build, and sidecar-smoke suites.

## Install

- **Desktop Lite (individuals):** download the installer for your platform from this release's assets, or follow [`docs/desktop-lite.md`](docs/desktop-lite.md) to build from source. A v0.1.8 install that was already wedged on the first-run loop will onboard normally after upgrading to v0.1.9 — no factory reset required.
- **Server (organizations):** see [`docs/prod-deploy.md`](docs/prod-deploy.md). Rebuild or re-pull the backend image to pick up the fix; no configuration changes are required.

## Known limitations in 0.1.9

- **Three breaking-major frontend upgrades held for dedicated migration:** Vite 8, Tailwind CSS 4, and ESLint 10 each require non-trivial migration work (Vite 8 config typing, Tailwind v4's config/engine rewrite, ESLint 9+ flat config) and are tracked as open PRs rather than rushed into this release. The esbuild advisory above is gated behind the Vite 8 upgrade.
- **pyo3 advisory suppressed, not resolved.** GHSA-36hh-v3qg-5jq4 lives in temporalio's vendored Rust bridge and cannot be fixed without an upstream temporalio release; the Trivy ignore is removed as soon as one ships against pyo3 ≥ 0.29. Carryover from v0.1.6.
- **No "rotate my codes" surface yet.** Carryover from v0.1.2.
- **macOS Intel installer is still not produced.** The CI matrix builds only `macos-latest` (Apple Silicon). Carryover from v0.1.1.
- **Desktop sidecar cold-start on Windows can still take 15–25 s** under the PyInstaller `--onefile` extract-to-temp path with active antivirus. Carryover from v0.1.3.
- **No E2E browser tests in CI yet** (#112) and **frontend/workflow coverage gaps** (#113) — carryover.

## Security

- After the wave, `pip-audit` and `pnpm audit --prod` report no known un-ignored vulnerabilities; the only documented suppressions remain CVE-2025-3000 (torch, local-only, no fixed release) and the temporalio-vendored pyo3 advisory carried in `.trivyignore` since v0.1.6. The js-yaml DoS is fixed and the esbuild advisory is dismissed as not applicable (see above).
- The first-run fix does not change the security posture: the bootstrap endpoint still requires no auth by design (it runs before any user exists), still returns `409` once any user exists, and the atomic single-insert TOCTOU guard is preserved — it now actually inserts on an empty database. The CORS allowlist, shutdown-token, parent-pid watchdog, Windows job-object cleanup, and async-safe graceful shutdown from v0.1.3 / v0.1.4 / v0.1.5 are unchanged.
- See [`docs/security.md`](docs/security.md) for the threat model and posture.
- Report vulnerabilities privately by emailing the maintainer — do not open a public issue.
