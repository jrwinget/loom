Fifth beta release of Loom — an evidence operating system for civil-rights legal teams that turns mixed field evidence (video, photos, audio, FOIA returns) into a defensible event timeline where every claim traces back to source material. Civil-liberties tooling: no face recognition, no suspicion scoring, no automated identity resolution.

> **Beta scope:** prepared for first use in Illinois (state and federal § 1983 actions). Defaults are documented in [`docs/requirements.md`](docs/requirements.md) and are jurisdiction-tunable.
>
> **Not legal advice.** Statutes and rules of evidence are referenced for design context only. Admissibility, consent, retention, and disclosure decisions for any specific matter belong with licensed counsel.

## Release focus

v0.1.4 is a corrective release for three desktop reliability bugs that left v0.1.3 installs unable to onboard a fresh user. The bundled webview could not reach its own sidecar; the lite-profile database shipped with no tables; and the Tauri shell's graceful-shutdown path was nesting a tokio runtime inside another, which would panic on `factory_reset` / `restart_backend`. The on-boot smoke test has been extended so the same class of bug cannot ship again silently. No new feature surfaces, no schema migrations.

## Highlights

- **The bundled frontend now actually talks to the sidecar.** Every API call from the webview was a relative URL (`/api/v1/...`). In `pnpm dev` the Vite proxy at `vite.config.ts` quietly rewrites those to `http://localhost:8000`, masking the problem in every developer workstation and every test run. In the production Tauri bundle the webview loads from `tauri://localhost`, and a relative URL resolves against the asset protocol — not the sidecar at `127.0.0.1:8000`. Every fetch from the production build 404d. The api-client now derives an absolute origin (`http://127.0.0.1:8000`) when `window.__TAURI_INTERNALS__` is present and stays relative everywhere else; the four raw `fetch` / `XMLHttpRequest` callers (`MfaSetup`, `use-assets`, `use-upload`, and the service-worker registration) flow through the same helper. The CSP at `tauri.conf.json` had whitelisted `connect-src http://127.0.0.1:8000` anticipating this absolute URL; the client now actually constructs it.
- **Fresh desktop installs materialise their own sqlite schema.** The sidecar's `__main__` used to call `uvicorn.run(...)` directly. A fresh install therefore ran every request against an empty `loom.db` with zero tables; once the URL bug above was fixed the first `/first-run/status` call would have 500d on `SELECT COUNT(*) FROM users`. The lite profile now runs `Base.metadata.create_all` from the bundled `loom.models` before uvicorn binds, and stamps `alembic_version` with the head revision so future migrations have a starting point. Gated on `LOOM_DEPLOYMENT_PROFILE == "lite"` so server-profile docker deploys retain `make migrate` as the authoritative migration entrypoint. The pyinstaller workflow now `--add-data`'s `alembic/` and `alembic.ini` into the binary so the script directory is resolvable at runtime via `sys._MEIPASS`.
- **Graceful sidecar shutdown no longer nests a tokio runtime.** `graceful_shutdown_sidecar` used `tauri::async_runtime::block_on` to perform the `/admin/shutdown` POST. Two of its callers (`factory_reset`, `restart_backend`) are `#[tauri::command] async fn`s, which means the `block_on` was running on a tokio worker thread that was already inside the runtime — that nesting panics with "Cannot start a runtime from within a runtime" on Tauri 2's multi-thread default. The body is now an `async fn` awaited directly from the async commands; a one-line sync wrapper retains `block_on` for the synchronous handlers (`ctrlc`, `on_window_event` CloseRequested, `RunEvent::Exit | ExitRequested`) where it is legal.
- **The "Reset Loom" escape hatch now also surfaces on /first-run.** The v0.1.3 hotfix put it on the login page's backend-unreachable branch with the rationale "the operator's only escape from a wedged install must not depend on the backend that is wedged." The same rationale applies one route over: after a factory reset the operator is redirected to `/first-run`, and if that page's status query errored they were left on a dead screen with no action. It now mirrors the login page's error branch — softened copy, a Reset Loom button inside Tauri, no button on the plain web profile.
- **The sidecar smoke test now covers `/first-run/status`.** `backend/scripts/smoke_sidecar.py` polled only `/api/v1/health`. A pyinstaller-built binary against an empty tempdir DB passes that check just fine — health does not touch the schema. The smoke now also fetches `/api/v1/first-run/status` and asserts `first_run_required: true` and `deployment_profile: "lite"`. That single extra assertion would have caught the schema-bootstrap regression in CI; it is the long-term guard against this class of "works in dev, breaks in installer" bug.

## Design notes

- **Why `metadata.create_all` instead of `alembic upgrade head` for lite first-launch?** The existing migration chain was authored against postgres. Several revisions call `op.add_column` with a `ForeignKey` constraint, which the sqlite dialect rejects without `batch_alter_table`. A fresh lite install has no history to preserve, so the bootstrap materialises the current model state in one shot and seeds `alembic_version` with the head revision directly via SQL — `command.stamp` is avoided because it re-execs `env.py` which calls `asyncio.run` and conflicts with the calling event loop when invoked from an async test fixture. Future migrations that touch lite-eligible models need to use `batch_alter_table`.
- **Why detect Tauri at call time rather than at module load?** Production webviews populate `window.__TAURI_INTERNALS__` before the bundle initialises, but tests in jsdom can flip it back and forth. Reading the flag inside `getApiOrigin()` keeps test isolation cheap and matches the existing pattern in `lib/tauri-bridge.ts`.
- **The async/sync `graceful_shutdown_sidecar` split is deliberate.** Both entry points share a single body — DRY across the five callers — but the legal use of `block_on` is from synchronous context only. The async commands `.await` the inner function directly; the sync handlers (panic-and-crash paths) drive the same `async fn` through a one-line `block_on` wrapper.

## Migrations

- No new Alembic revisions in v0.1.4. The lite profile now bootstraps its schema in-process on first launch and stamps `alembic_version` with the existing head; server-profile deploys continue to run `make migrate` out-of-band. Existing desktop installs that already onboarded are unaffected — `create_all` is a no-op when the tables already exist.

## Install

- **Desktop Lite (individuals):** download the installer for your platform from this release's assets, or follow [`docs/desktop-lite.md`](docs/desktop-lite.md) to build from source. Existing v0.1.2 / v0.1.3 installs upgrade in place — the bootstrap is idempotent. v0.1.3 installs that were stuck on the "Couldn't reach the backend" screen with no admin user will, after upgrading to v0.1.4, route correctly into onboarding on next launch.
- **Server (organizations):** see [`docs/prod-deploy.md`](docs/prod-deploy.md) for the production checklist (TLS, secrets, backup rotation, observability stack). No behavioural changes on the server profile; the schema bootstrap and the api-client origin override are both gated.

## Known limitations in 0.1.4

- **No "rotate my codes" surface yet.** Carryover from v0.1.2 — once you burn a recovery code you can still recover with the remaining seven, but you cannot mint fresh ones without factory-reset + re-bootstrap. Tracked for v0.1.5.
- **macOS Intel installer is still not produced.** The CI matrix builds only `macos-latest`, which resolves to Apple Silicon. Carryover from v0.1.1.
- **Desktop sidecar cold-start on Windows can still take 15–25 s** under the PyInstaller `--onefile` extract-to-temp path with active antivirus. Carryover from v0.1.3.
- **No E2E browser tests in CI yet** (#112) — the extended smoke covers `/first-run/status` against a real bundled binary, which catches the class of bug this release fixes, but a full webview-driven Playwright run is still tracked post-beta.
- **Frontend test coverage gaps** (#113) — carryover.
- **Workflow E2E coverage is partial** — carryover.

## Security

- The api-client origin override is gated on the runtime detection of `window.__TAURI_INTERNALS__`; a plain browser tab can never trigger the absolute-origin code path, so a hostile page cannot reach into a developer's running sidecar.
- The lite-profile schema bootstrap runs only when `LOOM_DEPLOYMENT_PROFILE == "lite"`. Server deployments retain external migration management; the bundled binary cannot mutate a postgres schema even if mis-deployed.
- The shutdown-token, parent-pid watchdog, and Windows job-object cleanup paths from v0.1.3 are unchanged in v0.1.4.
- See [`docs/security.md`](docs/security.md) for the threat model and posture.
- Report vulnerabilities privately by emailing the maintainer — do not open a public issue.
