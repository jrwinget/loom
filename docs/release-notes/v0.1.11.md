Tenth beta release of Loom — an evidence operating system for civil-rights legal teams that turns mixed field evidence (video, photos, audio, FOIA returns) into a defensible event timeline where every claim traces back to source material. Civil-liberties tooling: no face recognition, no suspicion scoring, no automated identity resolution.

> **Beta scope:** prepared for first use in Illinois (state and federal § 1983 actions). Defaults are documented in [`docs/requirements.md`](docs/requirements.md) and are jurisdiction-tunable.
>
> **Not legal advice.** Statutes and rules of evidence are referenced for design context only. Admissibility, consent, retention, and disclosure decisions for any specific matter belong with licensed counsel.

## Release focus

v0.1.11 fixes the first two defects reported after a desktop operator could finally onboard and reach the app: the **Cases** tab crashed to a blank screen, and signing back in after closing the app failed with "invalid email or password" despite correct credentials. Both turned out to be symptoms of one underlying problem — the frontend and the backend had drifted apart on the shape of API responses — plus a desktop-only database lookup bug that the catch-all login error had been hiding. This release repairs the class, not just the two screens.

## Highlights

- **The Cases tab no longer crashes (`frontend/src/hooks/use-case.ts`, `frontend/src/components/case/`).** The case list endpoint returns a paginated `{ items, total }` envelope, but the page treated the response as a plain array — so rendering the (empty) list threw and white-screened the app even before you had any cases. The page now unwraps the envelope, and a case with no description renders cleanly.
- **Signing back in after a restart works (`backend/src/loom/api/v1/auth.py`, `backend/src/loom/models/base.py`).** Three things combined to break the create-admin-then-sign-back-in flow on Desktop Lite, and only the first session (which auto-signs-in) hid them:
  - the email lookup was case-sensitive, so an address typed with different capitalization than at onboarding did not match — login now matches email case-insensitively and stores it normalized;
  - the profile lookup after sign-in (`/auth/me`) failed under the lite SQLite database because a user id taken from the session token is a string while the column is a UUID — UUID columns now accept string lookups, fixing this and every other id-keyed query under lite;
  - the sign-in screen reported **every** failure — network drop, server error, rate limit, even a post-login profile error — as "invalid email or password," which is what made this so hard to diagnose. It now distinguishes those cases.
  No factory reset is required: an admin created on an earlier build signs in normally after upgrading.
- **Response field names line up across the whole app (`frontend/src/lib/api-client.ts`).** The backend serializes fields in `snake_case`; the frontend expects `camelCase`. The API client now converts response keys in one place, so screens that read multi-word fields (timeline, assets, members, exports, and more) get the values they expect instead of blanks. Chain-of-custody manifests and other free-form records are passed through untouched.
- **Regression coverage added.** A real on-disk SQLite test drives create-admin → sign-in (same and differently-cased email) → `/auth/me`; new frontend tests cover the response transform, the case-list envelope, and the null-description render; and the desktop sidecar smoke test gained a fourth contract that exercises the full first-run → login → profile round-trip against the built binary.

## Migrations

- No new Alembic revisions and no schema changes in v0.1.11. Existing installs upgrade in place. New admin accounts store their email normalized to lowercase; existing accounts are matched case-insensitively, so no data migration or factory reset is needed.

## Install

- **Desktop Lite (individuals):** download the installer for your platform from this release's assets, or follow [`docs/desktop-lite.md`](docs/desktop-lite.md) to build from source. An install stuck behind the sign-in or Cases crash will work normally after upgrading — no factory reset required.
- **Server (organizations):** see [`docs/prod-deploy.md`](docs/prod-deploy.md). Rebuild or re-pull the backend image; no configuration changes are required.

## Known limitations in 0.1.11

- **Three breaking-major frontend upgrades held for dedicated migration:** Vite 8, Tailwind CSS 4, and ESLint 10 each require non-trivial migration work and are tracked as open PRs rather than rushed into this release. The esbuild advisory is gated behind the Vite 8 upgrade.
- **pyo3 advisory suppressed, not resolved.** GHSA-36hh-v3qg-5jq4 lives in temporalio's vendored Rust bridge and cannot be fixed without an upstream temporalio release; the Trivy ignore is removed as soon as one ships against pyo3 ≥ 0.29. Carryover from v0.1.6.
- **No "rotate my codes" surface yet.** Carryover from v0.1.2.
- **macOS Intel installer is still not produced.** The CI matrix builds only `macos-latest` (Apple Silicon). Carryover from v0.1.1.
- **Desktop sidecar cold-start on Windows can still take 15–25 s** under the PyInstaller `--onefile` extract-to-temp path with active antivirus. Carryover from v0.1.3.
- **No E2E browser tests in CI yet** (#112) and **frontend/workflow coverage gaps** (#113) — carryover. The contract-drift class is now guarded by the API-client transform tests plus the desktop sidecar smoke round-trip rather than a full browser E2E.

## Security

- No new dependencies in this release. The documented suppressions remain CVE-2025-3000 (torch, local-only, no fixed release) and the temporalio-vendored pyo3 advisory carried in `.trivyignore` since v0.1.6.
- The fixes do not weaken the security posture. Email is matched case-insensitively (already the de-facto expectation) and stored normalized; the login endpoint still returns a generic 401 for both unknown email and wrong password to avoid account enumeration; the UUID lookup change only accepts a string form of a value the server already trusts (its own signed token claim). The CORS allowlist, shutdown-token, parent-pid watchdog, and async-safe graceful shutdown are unchanged.
- See [`docs/security.md`](docs/security.md) for the threat model and posture.
- Report vulnerabilities privately by emailing the maintainer — do not open a public issue.
