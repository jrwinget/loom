Sixth beta release of Loom — an evidence operating system for civil-rights legal teams that turns mixed field evidence (video, photos, audio, FOIA returns) into a defensible event timeline where every claim traces back to source material. Civil-liberties tooling: no face recognition, no suspicion scoring, no automated identity resolution.

> **Beta scope:** prepared for first use in Illinois (state and federal § 1983 actions). Defaults are documented in [`docs/requirements.md`](docs/requirements.md) and are jurisdiction-tunable.
>
> **Not legal advice.** Statutes and rules of evidence are referenced for design context only. Admissibility, consent, retention, and disclosure decisions for any specific matter belong with licensed counsel.

## Release focus

v0.1.5 is a corrective release for the one desktop reliability bug v0.1.4 did not catch: the sidecar's CORS allowlist did not include the Tauri webview's actual origin, so even after v0.1.4's absolute-URL and schema-bootstrap fixes, every cross-origin fetch from the production webview to the sidecar was dropped by the browser engine before the response reached JavaScript. The login form's exception path swallowed the resulting `TypeError: Failed to fetch` as "Invalid email or password", which is what a tester would see regardless of the credentials they typed. The smoke test has been extended to exercise the preflight contract against the bundled binary so this class of allowlist drift cannot ship again silently. No new feature surfaces, no schema migrations.

## Highlights

- **The bundled webview can now actually call the sidecar.** v0.1.4 fixed the relative API URL and the missing lite-profile schema. What it missed: the FastAPI sidecar's default `cors_origins` was `["http://localhost:3000"]` only, and the config validator at `backend/src/loom/config.py` rejected any non-`http`/`https` scheme. The production Tauri 2 webview origin is `tauri://localhost` on macOS/Linux and `http://tauri.localhost` on Windows; `tauri dev` loads from `http://127.0.0.1:3000`, not `http://localhost:3000`. None of those three origins were in the allowlist. The validator now accepts the `tauri` scheme alongside `http`/`https`, and the default ships the four legitimate frontend origins (Vite dev under both spellings, plus the two webview origins). Server-profile deploys that override `LOOM_CORS_ORIGINS` are unaffected — the override replaces the list entirely. The CSP in `desktop/src-tauri/tauri.conf.json` already whitelisted `connect-src http://127.0.0.1:8000` from v0.1.3, so the frontend side did not change.
- **The sidecar smoke test now covers the preflight contract.** `backend/scripts/smoke_sidecar.py` hit `/api/v1/health` (since v0.1.0) and `/api/v1/first-run/status` (since v0.1.4). Neither sends an `Origin` header — urllib does not send one by default — so neither exercises the CORS middleware. The smoke now also issues `OPTIONS /api/v1/auth/login` with `Origin: tauri://localhost` and asserts the response echoes the origin in `Access-Control-Allow-Origin`. That single extra assertion would have caught the v0.1.4 leftover in CI; the same preflight is now asserted at the unit level in `backend/tests/unit/test_cors.py` via `httpx.ASGITransport` so the contract is locked at both the framework and the binary layer.

## Design notes

- **Why ship the Tauri origins in the default, not inject them from the Rust shell?** Tauri webview origins are deterministic constants of the framework, not deployment-dependent values. Putting them in `LOOM_CORS_ORIGINS` from `desktop/src-tauri/src/main.rs` would duplicate the constant across Rust + Python, require JSON-encoded env parsing on the Python side (pydantic-settings v2 does not split lists on commas), and silently drift if either side were updated without the other. Shipping them in the Python default keeps the source of truth single. The two `tauri.*` entries are inert under server-profile deploys because no public browser ever sends `Origin: tauri://localhost`; the allowlist matches by literal string and a value that never appears as an origin can never match.
- **Why the preflight test targets `/api/v1/auth/login` specifically.** It is the route the login form actually hits, it is a mutating endpoint (so it triggers a real preflight rather than a "simple request" that bypasses preflight), and it is on the CSRF middleware's exempt-paths list — so the CORS behavior is exercised in isolation without dragging the CSRF double-submit cookie into the test setup.

## Migrations

- No new Alembic revisions in v0.1.5. The lite profile continues to bootstrap its schema in-process on first launch (introduced in v0.1.4); server-profile deploys continue to run `make migrate` out-of-band. Existing desktop installs from v0.1.4 upgrade in place — the only on-disk artefact this release touches is the sidecar binary.

## Install

- **Desktop Lite (individuals):** download the installer for your platform from this release's assets, or follow [`docs/desktop-lite.md`](docs/desktop-lite.md) to build from source. Existing v0.1.4 installs upgrade in place. v0.1.4 installs that completed first-run onboarding but could not subsequently log back in are the precise target of this release.
- **Server (organizations):** see [`docs/prod-deploy.md`](docs/prod-deploy.md) for the production checklist (TLS, secrets, backup rotation, observability stack). The CORS default change is backwards-compatible with any server deploy that overrides `LOOM_CORS_ORIGINS`; deploys that relied on the v0.1.4 default inherit two harmless additional entries.

## Known limitations in 0.1.5

- **No "rotate my codes" surface yet.** Carryover from v0.1.2 — once you burn a recovery code you can still recover with the remaining seven, but you cannot mint fresh ones without factory-reset + re-bootstrap. Tracked for a later beta.
- **macOS Intel installer is still not produced.** The CI matrix builds only `macos-latest`, which resolves to Apple Silicon. Carryover from v0.1.1.
- **Desktop sidecar cold-start on Windows can still take 15–25 s** under the PyInstaller `--onefile` extract-to-temp path with active antivirus. Carryover from v0.1.3.
- **No E2E browser tests in CI yet** (#112) — the extended smoke covers the preflight contract against a real bundled binary, which catches the v0.1.4-class regression, but a full webview-driven Playwright run is still tracked post-beta.
- **Frontend test coverage gaps** (#113) — carryover.
- **Workflow E2E coverage is partial** — carryover.

## Security

- The CORS allowlist widening admits exactly the two Tauri webview origins and the two Vite dev-server origins. The field validator still rejects `*`, paths, empty entries, and every scheme other than `http`/`https`/`tauri`; widening to `tauri` is locked by an explicit unit test that confirms `file://`, `data:`, `chrome-extension://`, and `javascript:` remain rejected.
- The webview origin `tauri://localhost` is unreachable from a public browser; a hostile page on the internet cannot send `Origin: tauri://localhost` and have it honored, because the browser sets `Origin` from the loading document's URL, not from script.
- The shutdown-token, parent-pid watchdog, Windows job-object cleanup, and async-safe graceful-shutdown paths from v0.1.3 / v0.1.4 are unchanged in v0.1.5.
- See [`docs/security.md`](docs/security.md) for the threat model and posture.
- Report vulnerabilities privately by emailing the maintainer — do not open a public issue.
