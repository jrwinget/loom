Thirteenth beta release of Loom — an evidence operating system for civil-rights legal teams that turns mixed field evidence (video, photos, audio, FOIA returns) into a defensible event timeline where every claim traces back to source material. Civil-liberties tooling: no face recognition, no suspicion scoring, no automated identity resolution.

> **Beta scope:** prepared for first use in Illinois (state and federal § 1983 actions). Defaults are documented in [`docs/requirements.md`](docs/requirements.md) and are jurisdiction-tunable.
>
> **Not legal advice.** Statutes and rules of evidence are referenced for design context only. Admissibility, consent, retention, and disclosure decisions for any specific matter belong with licensed counsel.

## Release focus

v0.1.13 was the most usable desktop build yet, but two core evidence paths were broken on Desktop Lite: ingesting a URL returned `workflow service unavailable`, and a PDF upload sat on **pending** forever, then flipped to **Error** on "Upload all". The root cause was the same class of gap v0.1.13 fixed for navigation, one layer down: the documented Desktop Lite design — *"the same activity functions run in-process behind a thin Temporal-shaped facade … there is no Temporal server on Lite"* — was never actually built. Every workflow endpoint connected straight to a Temporal server that desktop installs don't run, so URL ingest, OCR, transcription, scene detection, and export all failed. Separately, the upload hooks posted to the wrong route and uploaded assets were never handed to the processing pipeline on any profile. This release builds the in-process facade, fixes the upload route contract, and wires uploads into processing. Backend + frontend; no schema changes.

## Highlights

- **Desktop Lite runs workflows in-process (`backend/src/loom/workflows/dispatch.py`, `sequences.py`, `lite_runner.py`).** A single dispatch gateway replaces the six copy-pasted `Client.connect(...)` blocks across the asset, OCR, transcript, scene, and export endpoints. On the server profile it starts a Temporal workflow exactly as before; on Lite it runs the workflow's activity sequence as a background task with no Temporal server. URL ingest and court-bundle export — whose dependencies are bundled with the desktop build — now work for real on Lite. The activity order and retry policy for both paths live in one place (`sequences.py`), so the server and in-process paths cannot drift.
- **Uploaded files are actually processed (`backend/src/loom/api/v1/assets.py`).** An upload created an asset with `processing_status="pending"` and dispatched nothing — on any profile — so it never advanced. Uploads now hand the asset to the ingest pipeline (hash verification → metadata → proxies → custody → complete) through the same gateway, so an uploaded PDF reaches **complete** instead of sitting on pending.
- **The upload route contract is fixed (`frontend/src/hooks/use-upload.ts`, `use-assets.ts`).** The upload hooks posted to `/cases/{id}/assets` but the endpoint is `/cases/{id}/assets/upload`, so every "Upload all" hit a route with no handler and flipped the file to **Error**. The asset-detail hook also fetched a non-existent `/assets/{id}` (404 — asset routes are case-scoped); it now uses the case-scoped path.
- **Workflow status and health are profile-aware (`backend/src/loom/api/v1/workflows.py`, `health.py`).** Status polling on Lite reports in-process progress, falling back to the rows a workflow produced; it no longer requires querying a Temporal server. `/health` no longer reports Lite as unhealthy for a Temporal server it deliberately doesn't run.
- **URL-ingest download moved to the profile-aware storage backend (`backend/src/loom/workflows/url_ingest_activities.py`).** The download activity still wrote through a hard-coded MinIO client — the one activity missed by the earlier local-storage migration — so even with the facade it would have failed on Lite. It now uses the same `get_storage_backend()` the other activities do.
- **Regression coverage added.** New tests lock the shared activity sequences to the behaviour they replaced, exercise the in-process runner and dispatch gateway, prove an uploaded PDF reaches **complete** on a real lite SQLite install with no Temporal, assert the workflow endpoints return *queued* (never 502) on Lite, and guard the frontend↔backend route contract that broke (`/assets/upload`, case-scoped detail).

## Migrations

- No Alembic revisions, schema, or configuration changes in v0.1.14. Existing installs upgrade in place with no data migration and no factory reset. Assets left stuck on **pending** by a prior build are reprocessed by re-uploading them; their original bytes and chain of custody are unaffected.

## Install

- **Desktop Lite (individuals):** download the installer for your platform from this release's assets, or follow [`docs/desktop-lite.md`](docs/desktop-lite.md) to build from source. No factory reset required.
- **Server (organizations):** see [`docs/prod-deploy.md`](docs/prod-deploy.md). Rebuild or re-pull both images. The server profile's Temporal-backed behaviour is unchanged.

## Known limitations in 0.1.14

- **AI-derived enrichment is empty on Desktop Lite without the models.** OCR, transcription, and scene detection now run in-process on Lite, but the desktop build does not bundle their AI dependencies (faster-whisper, pytesseract, scenedetect). These steps complete successfully with empty results rather than failing; install the server profile for full AI enrichment.
- **In-process workflows do not survive a force-quit.** A Lite ingest that is mid-flight when the app is force-quit is dropped (activities are idempotent, so re-uploading re-runs it cleanly). The server profile retains Temporal's durable retries.
- **Three breaking-major frontend upgrades held for dedicated migration:** Vite 8, Tailwind CSS 4, and ESLint 10. The esbuild advisory is gated behind the Vite 8 upgrade. Carryover from v0.1.13.
- **pyo3 advisory suppressed, not resolved.** GHSA-36hh-v3qg-5jq4 lives in temporalio's vendored Rust bridge and clears as soon as an upstream release ships against pyo3 ≥ 0.29. Carryover from v0.1.6.
- **No "rotate my codes" surface yet.** Carryover from v0.1.2.
- **macOS Intel installer is still not produced** (Apple Silicon only). Carryover from v0.1.1.
- **Desktop sidecar cold-start on Windows can still take 15–25 s.** Carryover from v0.1.3.
- **No E2E browser tests in CI yet** (#112) and **frontend/workflow coverage gaps** (#113) — carryover; this release's fixes are guarded by unit and integration tests.

## Security

- No new runtime dependencies and no posture change. The fixes are workflow dispatch, route contracts, and storage wiring. The documented suppressions remain CVE-2025-3000 (torch, local-only, no fixed release) and the temporalio-vendored pyo3 advisory carried in `.trivyignore` since v0.1.6.
- See [`docs/security.md`](docs/security.md) for the threat model and posture.
- Report vulnerabilities privately by emailing the maintainer — do not open a public issue.
