Security hotfix for v0.1.7 — an evidence operating system for civil-rights legal teams that turns mixed field evidence (video, photos, audio, FOIA returns) into a defensible event timeline where every claim traces back to source material. Civil-liberties tooling: no face recognition, no suspicion scoring, no automated identity resolution.

> **Beta scope:** prepared for first use in Illinois (state and federal § 1983 actions). Defaults are documented in [`docs/requirements.md`](docs/requirements.md) and are jurisdiction-tunable.
>
> **Not legal advice.** Statutes and rules of evidence are referenced for design context only. Admissibility, consent, retention, and disclosure decisions for any specific matter belong with licensed counsel.

## Release focus

v0.1.8 is a single-purpose security hotfix on top of v0.1.7, addressing a Starlette advisory (CVE-2026-54283) that was published immediately after the v0.1.7 cut. `request.form()` enforces its `max_fields` / `max_part_size` limits for `multipart/form-data` but silently ignored them for `application/x-www-form-urlencoded`, so an unauthenticated request with a very large field count or a single oversized field could block the event loop or force unbounded memory allocation — a denial-of-service vector. Starlette ships transitively via FastAPI; v0.1.7 resolved it at 1.3.0, which is affected. This release pins forward to the patched line. There are no other changes — no feature surfaces, no schema migrations, no API changes.

## Highlights

- **starlette 1.3.0 → 1.3.1.** Clears CVE-2026-54283 by forwarding and enforcing `max_fields` / `max_part_size` on the url-encoded form parser, matching the `multipart/form-data` defaults (`max_fields=1000`, `max_part_size=1MB`). Starlette is an indirect dependency (pulled by FastAPI; we declare no direct constraint), so the bump is confined to `backend/uv.lock`. The CORS, CSRF, rate-limit, and route/middleware paths that ride on Starlette are covered by the existing backend test suite and continue to pass unchanged.

## Migrations

- No new Alembic revisions in v0.1.8. No schema, data, or configuration changes. Existing v0.1.7 desktop installs upgrade in place; the only on-disk artefact this release touches is the sidecar binary, which now bundles the patched Starlette.

## Install

- **Desktop Lite (individuals):** download the installer for your platform from this release's assets, or follow [`docs/desktop-lite.md`](docs/desktop-lite.md) to build from source. Existing v0.1.7 installs upgrade in place.
- **Server (organizations):** see [`docs/prod-deploy.md`](docs/prod-deploy.md). Rebuild or re-pull the backend image to pick up the patched Starlette; no configuration changes are required. The default form limits (`max_fields=1000`, `max_part_size=1MB`) now apply to url-encoded bodies as well as multipart.

## Security

- Resolves CVE-2026-54283 (Starlette url-encoded form-parsing DoS, fixed in 1.3.1). After the bump, `pip-audit` reports no known un-ignored vulnerabilities; the only remaining documented suppressions are CVE-2025-3000 (torch, local-only, no fixed release) and the temporalio-vendored pyo3 advisory carried in `.trivyignore` since v0.1.6.
- This release stacks on the v0.1.7 `cryptography` 49.0.0 bump (OpenSSL advisory GHSA-537c-gmf6-5ccf). All other security posture from v0.1.6 / v0.1.7 — the CORS allowlist, shutdown-token, parent-pid watchdog, Windows job-object cleanup, and async-safe graceful shutdown — is unchanged.
- See [`docs/security.md`](docs/security.md) for the threat model and posture.
- Report vulnerabilities privately by emailing the maintainer — do not open a public issue.
