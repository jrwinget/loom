Fourth beta release of Loom — focused on two desktop reliability bugs reported against v0.1.2 that, together, could leave a beta tester locked out of their install with no in-app escape. The login screen now keeps its recovery affordances reachable through a transient backend hiccup, and the sidecar releases `127.0.0.1:8000` on every exit path so the next launch starts clean.

> **Beta scope:** unchanged from v0.1.1. Defaults are documented in [`docs/requirements.md`](docs/requirements.md) and remain jurisdiction-tunable.
>
> **Not legal advice.** See the v0.1.1 notes for the usual caveats.

## Highlights

- **Login recovery affordances stay visible during the boot window.** v0.1.2 derived "is this a Lite install?" from `useFirstRunStatus`, whose query was configured with `retry: false`. A single transient 5xx from `/first-run/status` while the backend lifespan was still warming up hid both "Forgot your password?" and "Reset Loom" until the operator force-quit the app. The query now inherits the QueryClient's default exponential backoff, the affordance block has explicit loading / error / success branches with stable layout, and "Reset Loom" stays reachable on the error branch inside the desktop shell — the operator's only escape from a wedged install cannot depend on the very backend that's wedged.

- **The desktop shell stops orphaning the sidecar on close.** The Tauri shell already called `child.kill()` on every shutdown path, but PyInstaller `--onefile` spawns the bootloader as the visible process and the real Python uvicorn as a child. Killing the bootloader left the Python child reparented to `init`/launchd, still holding `127.0.0.1:8000`, which forced the next launch into a 60-second "backend did not respond" timeout until the operator killed the stray process by hand. v0.1.3 covers all three exit paths in layers:

  1. **Graceful HTTP shutdown.** A new local-only `POST /api/v1/admin/shutdown` endpoint, authenticated by a per-launch shared secret (`LOOM_SHUTDOWN_TOKEN`), lets the desktop shell ask the sidecar to terminate cleanly. The Tauri shell mints the token at startup, hands it to the sidecar via env, and POSTs it back with a 1.5 s timeout on every cleanup path — window close, ctrl-c, panic, restart, factory-reset, and `RunEvent::Exit`. Server-profile deploys get the env unset, which masks the endpoint as a 404 — an external probe cannot tell the surface exists.
  2. **Unix orphan watchdog.** A daemon thread inside `loom-backend` polls `getppid()` and exits the moment the original parent dies. Gated on `LOOM_SHUTDOWN_TOKEN` being set, so direct `make dev` / `nohup` / `systemd` invocations don't self-terminate when their launching shell exits.
  3. **Windows job object.** Created at app setup with `JOB_OBJECT_LIMIT_KILL_ON_JOB_CLOSE`; every sidecar bootloader pid is assigned to the job after spawn. Closing the handle (which the OS does on process exit, including SIGKILL and force-quit) cascades termination to every in-job descendant. This is the safety net the graceful HTTP request cannot reach when the shell panics or is force-quit.

  `restart_backend` and `factory_reset` route through the graceful path so the respawn isn't racing a freshly-orphaned uvicorn for port 8000.

## Design notes

The three-layered approach for sidecar cleanup is deliberate. Each layer covers a different failure mode:

- **Graceful HTTP shutdown** is the happy path: it gives uvicorn a chance to drain in-flight requests and flush logs before exiting. It only works when the shell can run async code and the sidecar is responsive — i.e. the normal "user closes the window" case.
- **Parent-pid watchdog** covers the panic-or-crash case on Unix: the shell goes away without running cleanup, the sidecar's parent becomes pid 1, and the daemon thread notices within a second and exits.
- **Windows job object** covers the panic-or-crash case on Windows, where parent-pid semantics don't help (Windows reparents to nothing in particular). The job object is the kernel-side equivalent: when the shell's handle to the job is implicitly closed by process termination, the OS terminates every process still in the job.

The `pyinstaller --onefile` → `--onedir` migration mentioned as a v0.1.2 follow-up is deliberately still out of scope here. `externalBin` in `tauri.conf.json` only accepts single binaries, so a clean onedir migration is a non-trivial bundling refactor that doesn't fit a hotfix. The layered cleanup solves the orphan problem without touching the bundle layout.

## Migrations

- No new migrations in v0.1.3. The `password_recovery_codes` column added in v0.1.2 carries forward unchanged.

## Install

- **Desktop Lite:** as before. Existing v0.1.2 installs upgrade in place; there are no new database columns or filesystem changes. On upgrade, the desktop shell starts handing the sidecar a fresh `LOOM_SHUTDOWN_TOKEN` on every launch.
- **Server:** the `/admin/shutdown` endpoint exists in code on every profile but stays masked as a 404 wherever `LOOM_SHUTDOWN_TOKEN` is unset, which is the default for server deploys.

## Known limitations in 0.1.3

- **No "rotate my codes" surface yet.** Carried over from v0.1.2 — once you burn a recovery code you can still recover with the remaining seven, but you cannot mint fresh ones without factory-reset + re-bootstrap. Tracked for v0.1.4.
- **macOS Intel installer remains unbuilt** — the CI matrix carryover from v0.1.1 still applies.
- **`pyinstaller --onefile` cold-start latency on Windows persists.** The job-object fix releases port 8000 on close, but the 15–25 s extract-to-temp cost on first launch is still gated on the `--onedir` migration. The boot panel makes this visible without "Not Responding."
- All other v0.1.2 limitations carry forward.

## Security

- The shutdown endpoint is authenticated by a 64-character per-launch shared secret minted by the desktop shell (`secrets.token_hex(32)` equivalent) and never written to disk. It is compared with `secrets.compare_digest` to avoid timing leaks, and the endpoint masks as a 404 when the env var is unset so it doesn't broadcast its existence on server-profile deploys.
- The parent-pid watchdog and Windows job object are both opt-in (env-gated and Windows-only respectively); neither runs in a server-profile deploy.
- Report vulnerabilities privately by emailing the maintainer — do not open a public issue.
