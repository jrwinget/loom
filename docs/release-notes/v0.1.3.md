Fourth beta release of Loom — an evidence operating system for civil-rights legal teams that turns mixed field evidence (video, photos, audio, FOIA returns) into a defensible event timeline where every claim traces back to source material. Civil-liberties tooling: no face recognition, no suspicion scoring, no automated identity resolution.

> **Beta scope:** prepared for first use in Illinois (state and federal § 1983 actions). Defaults are documented in [`docs/requirements.md`](docs/requirements.md) and are jurisdiction-tunable.
>
> **Not legal advice.** Statutes and rules of evidence are referenced for design context only. Admissibility, consent, retention, and disclosure decisions for any specific matter belong with licensed counsel.

## Release focus

v0.1.3 closes two desktop reliability gaps reported against v0.1.2 that, together, could leave a beta tester locked out of their install with no in-app escape. The login screen now keeps its recovery affordances reachable through a transient backend hiccup during boot, and the sidecar releases `127.0.0.1:8000` on every shutdown path so the next launch starts clean. No new feature surfaces and no schema changes — this is a targeted reliability release.

## Highlights

- **Login recovery affordances stay visible during the boot window.** v0.1.2 derived "is this a Lite install?" from `useFirstRunStatus`, whose query was configured with `retry: false`. A single transient 5xx from `/first-run/status` while the backend lifespan was still warming up hid both "Forgot your password?" and "Reset Loom" permanently — no spinner, no error state — until the operator force-quit the app. The query now inherits the QueryClient's default exponential backoff, the affordance block has explicit loading / error / success branches with stable layout, and "Reset Loom" stays reachable on the error branch inside the desktop shell. The operator's only escape from a wedged install cannot depend on the very backend that's wedged. Test coverage in `login-recovery-affordances.test.tsx` grows from 2 to 6 cases (loading, error+tauri, error+web, all the existing happy paths).
- **The desktop shell stops orphaning the sidecar on close.** The Tauri shell already called `child.kill()` on window close, ctrl-c, panic, restart, factory-reset, and `RunEvent::Exit`. But PyInstaller `--onefile` spawns the bootloader as the visible process and the real Python uvicorn as a child. Killing the bootloader left the Python child reparented to `init`/launchd, still holding `127.0.0.1:8000`, which forced the next launch into a 60-second "backend did not respond" timeout until the operator killed the stray process by hand. v0.1.3 covers all three exit paths in layers, summarised in the design note below.
- **Local-only `POST /api/v1/admin/shutdown` endpoint** for graceful sidecar termination, authenticated by a 64-character per-launch shared secret (`LOOM_SHUTDOWN_TOKEN`). The desktop shell mints the token at startup, hands it to the sidecar via env, and POSTs it back with a 1.5 s timeout on every cleanup path. Server-profile deploys get the env unset, which masks the endpoint as a 404 — an external probe cannot tell the surface exists.

## Design notes

The three-layered approach for sidecar cleanup is deliberate. Each layer covers a different failure mode and the layers cannot be collapsed without dropping coverage of one of them:

- **Graceful HTTP shutdown** is the happy path: it gives uvicorn a chance to drain in-flight requests and flush logs before exiting. The endpoint schedules `os._exit(0)` ~100 ms after returning 204 so uvicorn can write the response back out. This layer only works when the shell can run async code and the sidecar is responsive — the normal "user closes the window" case.
- **Unix parent-pid watchdog** covers the panic-or-crash case on Unix: a daemon thread in `loom-backend` polls `os.getppid()` once a second and exits the moment the original parent becomes pid 1. It is gated on `LOOM_SHUTDOWN_TOKEN` being set, so direct `make dev` / `nohup` / `systemd` invocations get the env unset and skip the watchdog — a backgrounded server doesn't self-terminate when its launching shell exits.
- **Windows job object** covers the panic-or-crash case on Windows, where parent-pid semantics don't help. The job is created at app setup with `JOB_OBJECT_LIMIT_KILL_ON_JOB_CLOSE`; every sidecar bootloader pid is assigned to it after spawn. When the shell's handle to the job is implicitly closed by process termination (including SIGKILL and force-quit), the OS terminates every process still in the job. This is the safety net the graceful HTTP request cannot reach.

`restart_backend` and `factory_reset` both route through the graceful path so the respawn isn't racing a freshly-orphaned uvicorn for port 8000.

The `pyinstaller --onefile` → `--onedir` migration noted as a v0.1.2 follow-up is deliberately still out of scope. `externalBin` in `tauri.conf.json` only accepts single binaries, so a clean onedir migration is a non-trivial bundling refactor that doesn't fit a hotfix. The layered cleanup solves the orphan problem without touching the bundle layout.

## Install

- **Desktop Lite (individuals):** download the installer for your platform from this release's assets, or follow [`docs/desktop-lite.md`](docs/desktop-lite.md) to build from source. Existing v0.1.2 installs upgrade in place — there are no new database columns or filesystem changes. On every launch the desktop shell now hands the sidecar a fresh `LOOM_SHUTDOWN_TOKEN`. Single-user, local-only — nothing leaves the machine unless you export.
- **Server (organizations):** see [`docs/prod-deploy.md`](docs/prod-deploy.md) for the production checklist (TLS, secrets, backup rotation, observability stack). The `/admin/shutdown` endpoint exists in code on every profile but stays masked as a 404 wherever `LOOM_SHUTDOWN_TOKEN` is unset, which is the default for server deploys.

## Known limitations in 0.1.3

- **No "rotate my codes" surface yet.** Carryover from v0.1.2 — once you burn a recovery code you can still recover with the remaining seven, but you cannot mint fresh ones without factory-reset + re-bootstrap. Tracked for v0.1.4.
- **macOS Intel installer is still not produced.** The CI matrix builds only `macos-latest`, which resolves to Apple Silicon. Carryover from v0.1.1.
- **Desktop sidecar cold-start on Windows can still take 15–25 s** under the PyInstaller `--onefile` extract-to-temp path with active antivirus. The job-object fix releases port 8000 on close, but the first-launch extract cost is gated on the `--onedir` migration. The boot panel makes this visible without "Not Responding."
- **No E2E browser tests in CI yet** (#112) — Playwright smoke flow is tracked post-beta; golden-path browser flows continue to be exercised manually.
- **Frontend test coverage gaps** (#113) — roughly 14 hooks and 11 components without dedicated tests, tracked post-beta.
- **Workflow E2E coverage is partial** — ingest is exercised end-to-end (#119); correlation, scene, OCR, transcription, export, and url-ingest workflows still rely on structural assertions (#113).

## Security

- The shutdown endpoint is authenticated by a 64-character per-launch shared secret minted by the desktop shell (`secrets.token_hex(32)`) and never written to disk. It is compared with `secrets.compare_digest` to avoid timing leaks, and the endpoint masks as a 404 when the env var is unset so it doesn't broadcast its existence on server-profile deploys.
- The parent-pid watchdog and Windows job object are both opt-in: the watchdog is env-gated on `LOOM_SHUTDOWN_TOKEN`, the job object is Windows-only and only ever holds sidecar pids the shell itself spawned. Neither runs in a server-profile deploy.
- See [`docs/security.md`](docs/security.md) for the threat model and posture.
- Report vulnerabilities privately by emailing the maintainer — do not open a public issue.
- One outstanding upstream advisory: CVE-2026-3219 (`pip` ≤ 26.0.1, no upstream patch yet) remains documented as tolerable_risk; pip is build-time only and supply-chain attacks on the upstream toolchain are out-of-scope per the threat model.
