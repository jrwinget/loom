Ninth beta release of Loom — an evidence operating system for civil-rights legal teams that turns mixed field evidence (video, photos, audio, FOIA returns) into a defensible event timeline where every claim traces back to source material. Civil-liberties tooling: no face recognition, no suspicion scoring, no automated identity resolution.

> **Beta scope:** prepared for first use in Illinois (state and federal § 1983 actions). Defaults are documented in [`docs/requirements.md`](docs/requirements.md) and are jurisdiction-tunable.
>
> **Not legal advice.** Statutes and rules of evidence are referenced for design context only. Admissibility, consent, retention, and disclosure decisions for any specific matter belong with licensed counsel.

## Release focus

v0.1.10 finishes the desktop first-run reliability work. v0.1.9 fixed the 409 redirect loop that blocked onboarding outright, but a fresh install could still fail to create its admin account on the **first** attempt: the account creation showed a bare "load failed", a second click then worked, and onboarding could bounce back to the welcome screen afterwards. These were three distinct, residual defects in the first-run path — none of them the v0.1.9 SQL fix. This release closes all three and adds regression coverage.

## Highlights — desktop first-run reliability

- **First submit no longer fails with "load failed" (`backend/src/loom/__main__.py`, `frontend/src/lib/api-client.ts`).** The desktop webview opens an HTTP keep-alive socket on the initial `GET /first-run/status` and reuses it for the account-creation `POST`. The sidecar's uvicorn ran with the default 5-second keep-alive idle timeout, so while the operator filled out the admin form (comfortably more than 5 seconds) the sidecar closed the idle socket. The reused-socket `POST` was then written to a half-closed connection and reset at the transport layer — `fetch()` rejected with a `TypeError`, surfaced by WebKit as the bare string "Load failed". Because this is a transport reject (no HTTP response was ever received), the request was never processed by the server, so a retry on a fresh connection is safe. The API client now retries a transport-level fetch rejection up to twice with a short backoff; HTTP error statuses (4xx/5xx) are never retried. The sidecar's keep-alive idle window is also widened to cover normal form-fill time as defense in depth.
- **Onboarding no longer bounces back to the welcome screen (`frontend/src/hooks/use-first-run.ts`).** The first-run status query is cached for 60 seconds, and completing onboarding did not refresh it. After creating the admin and routing into the app, the first-run guard read the stale "no users yet" value and redirected back to onboarding. Completing first-run now updates the cached status to "complete" synchronously, so the guard lets the new admin through.
- **Choosing a custom data directory keeps the admin you create (`frontend/src/routes/first-run.tsx`).** Previously the admin was created against the *current* data directory, and the sidecar was only restarted onto the newly chosen directory *afterwards* — landing the operator in a fresh, empty database with the just-created admin discarded, re-triggering first-run. The directory switch (and sidecar restart) now happens **before** account creation: when the operator confirms a different directory, the shell restarts the sidecar onto it and waits for the new instance to pass its health check, so the bootstrap admin is written into the final database.
- **Regression coverage added.** New frontend tests cover the transport-retry behavior (retries a network reject, never retries 4xx/5xx, preserves the 401 sign-out path), the first-run status cache update on completion, and the directory-switch-before-account-creation ordering. A desktop unit test pins the respawn to the persisted data directory.

## Migrations

- No new Alembic revisions in v0.1.10. No schema, data, or configuration changes. Existing installs upgrade in place; the only runtime-affecting changes are the first-run client/sidecar reliability fixes, which take effect only on a not-yet-onboarded install.

## Install

- **Desktop Lite (individuals):** download the installer for your platform from this release's assets, or follow [`docs/desktop-lite.md`](docs/desktop-lite.md) to build from source. An install still stuck on the v0.1.9 first-run path will onboard normally after upgrading — no factory reset required.
- **Server (organizations):** see [`docs/prod-deploy.md`](docs/prod-deploy.md). Rebuild or re-pull the backend image to pick up the keep-alive change; no configuration changes are required.

## Known limitations in 0.1.10

- **Three breaking-major frontend upgrades held for dedicated migration:** Vite 8, Tailwind CSS 4, and ESLint 10 each require non-trivial migration work and are tracked as open PRs rather than rushed into this release. The esbuild advisory is gated behind the Vite 8 upgrade.
- **pyo3 advisory suppressed, not resolved.** GHSA-36hh-v3qg-5jq4 lives in temporalio's vendored Rust bridge and cannot be fixed without an upstream temporalio release; the Trivy ignore is removed as soon as one ships against pyo3 ≥ 0.29. Carryover from v0.1.6.
- **No "rotate my codes" surface yet.** Carryover from v0.1.2.
- **macOS Intel installer is still not produced.** The CI matrix builds only `macos-latest` (Apple Silicon). Carryover from v0.1.1.
- **Desktop sidecar cold-start on Windows can still take 15–25 s** under the PyInstaller `--onefile` extract-to-temp path with active antivirus. Carryover from v0.1.3.
- **No E2E browser tests in CI yet** (#112) and **frontend/workflow coverage gaps** (#113) — carryover. The transport-level keep-alive race in particular is not reproducible under the unit-test HTTP transport (it has no socket layer); it is covered by the client-retry unit tests plus the desktop sidecar-smoke and a manual first-run smoke.

## Security

- No new dependencies in this release. The documented suppressions remain CVE-2025-3000 (torch, local-only, no fixed release) and the temporalio-vendored pyo3 advisory carried in `.trivyignore` since v0.1.6.
- The first-run fixes do not change the security posture: the bootstrap endpoint still requires no auth by design (it runs before any user exists), still returns `409` once any user exists, and the atomic single-insert TOCTOU guard from v0.1.9 is preserved. The transport-retry only replays requests that received no response, and never retries an authenticated 4xx/5xx. The CORS allowlist, shutdown-token, parent-pid watchdog, Windows job-object cleanup, and async-safe graceful shutdown are unchanged.
- See [`docs/security.md`](docs/security.md) for the threat model and posture.
- Report vulnerabilities privately by emailing the maintainer — do not open a public issue.
